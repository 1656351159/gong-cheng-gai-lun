# 材智通——智慧材料库数字化管理系统需求规格说明书

## 3.1 引言

### 3.1.1 编写目的
本需求规格说明书用于描述“材智通——智慧材料库数字化管理系统”样机版的软件需求，作为后续概要设计、详细设计、编码实现和测试的依据。读者包括项目组成员、指导教师和未来的维护人员。

### 3.1.2 项目范围
本项目样机版聚焦实验室/工场材料管理的核心流程，实现材料类目管理、材料入库、材料领用、库存查询及基础数据导出等功能，支持多角色权限控制，用于验证整体业务方案的可行性。

### 3.1.3 术语与缩略语
- **材料**：实验或教学过程中使用的耗材、工具等。
- **入库**：材料从采购到进入库存的登记过程。
- **领用**：师生从材料库领取材料的登记过程。
- **库存**：某种材料当前可用数量。
- **管理员**：负责系统配置与权限分配的角色。

## 3.2 总体描述

### 3.2.1 系统定位
系统定位为校园实验室材料管理平台的一期样机，采用 B/S 架构，在单机环境下完成部署和演示。样机的目标是支撑课程答辩演示及文档编写，不直接用于生产环境。

### 3.2.2 用户角色
1. **游客**：无账号，仅能查看登录页和简单说明。
2. **学生**：登录后可以查询材料信息、提交领用申请、查看个人领用记录。
3. **实验员**：负责材料入库、库存盘点、处理领用申请。
4. **管理员**：负责用户管理、材料类目维护、系统参数配置。

> 若时间有限，可只实现学生、实验员、管理员三类角色。

### 3.2.3 运行环境
- **客户端**：Chrome/Edge 浏览器，1920×1080 分辨率优先。
- **服务器端**：一台本地 Windows / Linux 笔记本或台式机。
- **开发与运行语言**：后端采用 Java Spring Boot 或等价 Web 框架；前端采用 React 或基于 HTML+JS 的简化实现；数据库使用 PostgreSQL / MySQL / SQLite 之一。
- **部署方式**：本地部署，支持通过脚本或 Docker-Compose 一键启动（选其一即可）。

### 3.2.4 设计约束与假设
- 本项目为课程样机，优先确保核心业务流程可用，性能目标为单机少量并发用户。
- 目前不接入学校统一身份认证，用户信息在系统内部维护。
- 不考虑真实财务系统对接，仅提供格式相似的导出功能。

## 3.3 功能需求
以下用 FR 编号列出样机需要实现的主要功能（FR1–FR6 为基础，FR7/FR8 视精力而定）。

### FR1 登录与权限控制
- **FR1-1**：系统提供登录页面，用户输入账号和密码可以登录。
- **FR1-2**：系统根据用户角色（学生 / 实验员 / 管理员）控制菜单与操作权限。
- **FR1-3**：未登录用户访问业务页面时自动跳转到登录页。
- **FR1-4**：管理员可以在后台创建、修改、禁用用户账户。

### FR2 材料类目与材料信息管理
- **FR2-1**：管理员可以维护材料类目（如：化学试剂 / 电子元器件等），支持至少三级分类。
- **FR2-2**：管理员可以为每种材料录入名称、规格型号、计量单位、单价等基础信息。
- **FR2-3**：系统提供材料列表页面，支持按名称、类目关键字搜索和分页浏览。

### FR3 材料入库管理
- **FR3-1**：实验员可以新增入库记录，填写材料、数量、入库日期、供应商等信息。
- **FR3-2**：提交入库记录后，系统自动增加对应材料的库存数量。
- **FR3-3**：系统提供入库记录列表，可按时间范围、材料名称查询。

> 若要扩展扫码入库，可额外增加条码字段的录入需求。

### FR4 材料领用管理
- **FR4-1**：学生登录后可以发起领用申请，填写材料、数量、用途、课程/项目号等字段。
- **FR4-2**：实验员在后台审核领用申请（通过/驳回），通过后自动扣减库存。
- **FR4-3**：当领用数量超过现有库存时，系统提示错误并禁止提交。
- **FR4-4**：学生可以在“我的领用”页面查看历史记录。

### FR5 库存查询与预警（预警可做简化版）
- **FR5-1**：实验员和管理员可以在“库存管理”页面查看各材料当前库存数量。
- **FR5-2**：支持按类目、关键字搜索，并显示总库存、单位等信息。
- **FR5-3（可选）**：管理员可以为每种材料设置安全库存阈值；当库存低于阈值时，在列表中以高亮/标记方式提醒。

### FR6 报表导出
- **FR6-1**：管理员可以按照时间范围导出“领用明细表”（包含材料名称、数量、使用人、用途、时间等字段），导出格式为 Excel 或 CSV。
- **FR6-2**：管理员可以导出“当前库存表”，用于期末统计。

### FR7 系统管理（可选强化）
- **FR7-1**：管理员可以配置系统基本信息（系统名称、页脚说明等）。
- **FR7-2**：管理员可以查看简单的统计信息，如材料总数、领用次数 Top N 等。

## 3.4 非功能需求（样机版）

### 3.4.1 性能需求
- 在单机部署环境下，同时有 3–5 个用户进行基本操作时，页面响应时间不超过 2 秒。

### 3.4.2 安全性
- 登录密码以加密方式存储，不以明文出现在数据库中。
- 关键操作（入库、领用、删除记录）需要登录状态，并按角色控制权限。

### 3.4.3 可用性
- 常用操作（登录、查询、入库、领用）不超过 3 步即可完成。
- 提供错误提示和必填项校验，防止用户漏填关键字段。

### 3.4.4 可维护性
- 前后端代码按模块划分清晰，类和接口命名规范。
- 使用 README 或部署文档说明项目结构和启动方式，方便后续同学维护。

## 4. 本地部署准备
答辩要求本地部署，需要在代码和工程方面做好以下准备：

1. **一键启动脚本 / 文档**
   - 在项目根目录准备 `run.bat`（Windows）或 `run.sh`（Linux），脚本依次完成：
     1. 启动数据库（如使用 Docker 可执行 `docker-compose up`；若使用 SQLite/H2 可省略）。
     2. 启动后端服务。
     3. 启动前端（例如 `npm start` 或运行打包后的静态文件）。
   - 编写 `部署说明.md`，列出所需环境（JDK、Node、数据库版本）和启动步骤。

2. **简化数据库方案**
   - 优先考虑 H2/SQLite 等嵌入式数据库，便于在老师电脑上直接运行。
   - 若使用 MySQL/PostgreSQL，需要提供安装与建库脚本，并写明步骤。

3. **预置演示数据**
   - 准备 SQL 或初始化脚本，自动插入材料、用户、入库/领用记录等信息，确保启动后即可演示。

4. **离线可运行**
   - 所有依赖的 JS/CSS 库尽量本地化，避免使用 CDN。
   - 不依赖外部 API；例如预警可通过页面高亮提示，无需真实发送消息。

5. **演示账号清单**
   - 预设并记录常用账号，答辩时可按顺序演示：
     - 管理员：admin / 123456
     - 实验员：lab / 123456
     - 学生：stu / 123456
